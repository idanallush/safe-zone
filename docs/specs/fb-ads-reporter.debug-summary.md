## Performance Notes

- **`useColumnConfig` refactor**: Removing the `useEffect` + `setState` pattern and using a `useState` lazy initializer eliminates a render cycle on mount. Previously the component rendered with `DEFAULT_VISIBLE_METRICS`, then re-rendered with localStorage values. Now it reads localStorage synchronously during initialization â€” one fewer render.

- **`adsFetcher` extraction**: Moving the SWR fetcher outside the hook prevents React Compiler from receiving a new function reference on every render. This is low-impact now but important for future Compiler optimization passes.

- **TanStack Table warning**: The React Compiler skips memoizing `CreativeTable` because of `useReactTable`. In practice this means the table may re-render slightly more than necessary. To resolve fully, you would need to either (a) add `// eslint-disable-next-line react-hooks/incompatible-library` and accept the behavior, or (b) upgrade to TanStack Table v9 when it stabilizes with React Compiler support.

- **No virtual scrolling**: For large accounts (1000+ ads), the grid renders all cards into the DOM. The integration summary notes the 1000-ad pagination cap. If this cap is ever raised, virtual scrolling (e.g., `@tanstack/react-virtual`) should be added to `AdGrid`.

- **SWR deduplication**: The `dedupingInterval: 30000` (30 seconds) on ad data fetching is appropriate for an internal reporting tool.

## Remaining Concerns

1. **Storage race condition**: If two PDF exports are triggered simultaneously, both will `readIndex()` â†’ modify â†’ `writeIndex()` concurrently, and the last write wins (one report may be lost from the index). The PDF file itself will still be on disk. For a single-team tool with low concurrency this is acceptable, but a file-based lock or database should be added before multi-user rollout.

2. **TanStack Table React Compiler warning**: Not fixable without switching libraries. Does not affect correctness.

3. **`useFacebookAds` hook duplication**: There is a `useFacebookAds` hook in `src/hooks/use-facebook-ads.ts` that is not used anywhere â€” `ads/page.tsx` defines its own `useAds` inline. The standalone hook could be removed to reduce confusion, or the page should be refactored to use it. Not a bug, flagged for cleanup.

4. **Facebook token in API logs**: The Facebook access token appears in server-side fetch URLs (e.g., `https://graph.facebook.com/v25.0/.../ads?...&access_token=EAABb...`). This is Facebook's own API pattern and unavoidable with their current API design. Ensure server logs are not stored long-term or are access-controlled.

5. **No request size limits on PDF routes**: The `POST /api/pdf/quick` and `POST /api/pdf/client` routes accept an unlimited JSON body. A large payload (e.g., thousands of ads with full copy text) could cause memory pressure. Consider adding a `body-size-limit` or validating `ads.length` against a maximum.

6. **`creativeTable` component appears unused**: `src/components/dashboard/creative-table.tsx` defines a full TanStack Table implementation but `ads/page.tsx` uses `AdGrid` (card view) exclusively. The table view may be intended for a future tab. It compiles correctly but is currently dead code.
